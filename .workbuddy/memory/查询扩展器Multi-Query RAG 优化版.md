## 查询扩展器（Multi-Query RAG 优化版）
 核心思路：使用低成本模型生成 3~5 个查询变体，并行检索扩大召回覆盖，
 再经过去重、重排序、截断，将高价值文档注入 Prompt，提升答案质量并控制成本。

 作者: jsq (优化版)

 详细步骤与细节把控：

 1. 查询扩展与并行检索
    - 配置一个便宜的 LLM（如 GPT-3.5、本地 7B 模型），根据原始查询生成 3~5 个语义接近但角度不同的变体。
      示例：原问题“如何优化数据库查询？”可扩展为“减少慢查询的方法”“索引优化策略”等。
    - 将原始查询与所有扩展查询一起，异步并发发送给 DocumentRetriever（避免串行阻塞），每个查询返回 Top-K 个文档片段（K 建议 5~10）。
    - 注意：检索延迟主要取决于向量数据库，并发可显著压缩总耗时，但需限流防止打爆检索服务。

 2. 文档融合与去重
    - 收集所有检索结果（原始查询 + 变体查询的结果），形成一个候选文档池。
    - 去重策略（按优先级选择）：
        a) 优先根据文档唯一标识（如 doc_id、URL、数据库主键）去重。
        b) 若无唯一 ID，采用文本相似度哈希（如 SimHash、MinHash）计算签名，合并相似度 > 阈值（如 0.9）的文档。
        c) 不推荐纯字符串精确匹配，因为同一内容可能被切分为不同片段。
    - 去重时，每个文档保留其在多个查询中的最高检索得分，供后续排序参考。

 3. 相关性重排序与智能截断
    - 针对去重后的文档，使用一个重排序模型（如 bge-reranker、Cohere Rerank）或直接调用便宜 LLM，
      计算每篇文档与【原始查询】（而非扩展变体）的相关性分数。
    - 按分数降序排列，取前 N 篇（N 的取值需要动态计算，见下一步）。
    - 截断细节：
         估算模型最大上下文长度（如 4096 token），减去系统提示和预留输出长度（如 1024 token），
          剩余空间即为可用文档 token 配额。
         从最高分文档开始累加 token 数，直到达到配额的 80%~90%，丢弃后续文档。
         若某单篇文档过长，可截取最相关的段落（例如根据与查询的句子相似度取前 512 token），
          或调用小模型生成文档摘要（但需权衡信息丢失风险）。

 4. Prompt 改写与上下文注入
    - 构建最终 Prompt，结构清晰分层：
        [系统指令] 明确角色与规则，如“你是一个技术助手，请严格根据提供的文档回答用户问题。”
        [参考文档] 使用特殊标记包裹每篇文档，例如：
            [DOC1] 文档内容... [/DOC1]
            [DOC2] 文档内容... [/DOC2]
        [用户问题] 原始查询
    - 在系统指令中加入防幻觉约束：“如果文档信息不足，请明确告知无法回答，不要编造内容。”
    - 调用 LLM 前，使用 token 计数器（如 tiktoken）再次校验总 token 数，确保不超限，否则自动按上述截断规则回退。

 性能与成本优化关键点：
    - 扩展模型选用便宜模型，单次成本极低，且可设置缓存（对高频相似查询的扩展结果复用），进一步节省调用量。
    - 检索与扩展生成可并行：在查询扩展的同时，原始查询可先发起首批检索，实现流水线加速。
    - 记录每次请求的检索命中效果（最终答案是否采纳了文档、用户反馈），持续优化扩展数量 K 与保留文档数 N。
    - 当用户问题很短或很明确时，可跳过扩展步骤，直接单查询检索，避免不必要的开销。
